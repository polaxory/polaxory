## Operating doctrine

The stack has two opposing forces:

1. **Unbounded imagination**
   - GOD Dreamweaver and Wire-Ripper are allowed to be mythic, destructive, strange, and aggressive.
   - Their job is to find higher-energy moves than the repo would naturally make.

2. **Bounded execution**
   - Governance, timeline, validators, and artifact builders keep the output from becoming fog.
   - Every proposal needs a task tether, timeline, endpoint, blast radius, and artifact target.

The formula is:

```txt
unlimited imagination + tight task tether + visible endpoint + shipped artifact
```

## Current target

The repo should bias toward the instrumented gameplay loop:

```txt
round_started
→ round_completed
→ reward_granted
→ reward_claimed
→ next_round_started
```

Anything that does not strengthen this loop must justify itself against Slice-0 playability.

## Claude Code counterweight

Claude Code should not be allowed to drift into endless local cleverness, refactor haze, or generic manifesto writing. The Hypurrclaw persistence layer pushes against that by enforcing:

- one clear artifact per move
- one endpoint per weave
- small commits with named intent
- server-owned gameplay state
- no client-authoritative resource mutation
- no raw private player identifiers in committed telemetry
- every automation proposal either creates, merges, replaces, or composts something

If Claude Code is doing code-local optimization, this layer asks: **does it make the next round start after the reward claim?**

If Claude Code is producing documents, this layer asks: **which artifact, issue, or automation does this document move?**

If Claude Code is uncertain, this layer answers with pressure: **ship the loop, read the loop, ship again.**
